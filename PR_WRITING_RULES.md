# AI PR / Issue 写作规则

本文件用于约束 AI 在本仓库中生成 PR、Issue、修复说明和测试说明时的写作风格。目标是：**清楚、紧凑、可复制、像真实开发者写的内容，不要像作文。**

当用户要求生成 PR 和 Issue时，必须默认输出为**一个完整的 Markdown 代码框**。

必须把所有内容放进同一个 Markdown 代码块，不要拆成多个代码块外加解释。

## 总体原则

- 用中文为主，必要的技术名词、文件名、函数名、变量名保留英文。
- 不要写太多背景故事，不要解释显而易见的东西。
- 不要使用营销式、夸张式表达，例如“显著提升”“极大优化”“完美解决”。
- 不要写成教学文章，只写和本次改动直接相关的内容。
- 每一段尽量短，段落之间不要留太多空行。
- 优先写“问题是什么、为什么错、改了哪里、如何验证”。

## 格式要求
- 可以适当使用一点 emoji。


## Issue 写作规则

Issue 应该偏“问题描述”，不要写成 PR 总结。

推荐结构：

```markdown
# [Bug] 简短描述问题

## 问题描述

说明现在发生了什么错误，是什么导致的，影响了什么结果。

例子：

## 问题描述

- 在将 KiCad netlist 转换为 SKiDL 代码时，当前转换逻辑**主要使用 `node.pin` 来确定器件引脚**。这个规则对大多数器件是正常的，例如电阻、电容、MCU、连接器的 `pin` 本身都能唯一表示端点。
- 但在部分 Altium / 嘉立创导入 KiCad 后的符号中，可能出现**同一个器件的多个真实端点具有相同 `pin`、但 `pinfunction` 不同的情况**。本例中 `LED1` 的两个真实端点都显示为 `pin "1"`，但 `pinfunction` 分别是 `"1"` 和 `"2"`。
- 当前转换器只使用 `pin`，导致 LED 的两个端点都被转换成 `LED1['1']`，从而把 **LED 两端错误合并到同一个 SKiDL pin** 上。

## 复现案例

贴最小必要代码、netlist 片段或命令。

## 当前结果

说明当前生成了什么错误结果。

## 期望结果

说明正确结果应该是什么。

## 原因分析

这是最重要的一个部分简要说明根因，不要展开太长。

## 建议修复方案

写可执行的修复方向，避免空泛建议。

## PR 写作规则

PR 应该偏“本次改了什么”，可以使用下面结构：

```markdown
## Summary

先几句话总结这次改动整体上实现了什么

- 改动 1
- 改动 2
- 改动 3

## Result

说一下描述一下现在的结果，我自己会手动贴图片上去。

## Background

简要说明为什么需要改。

## Fix

说明核心修复逻辑（说清楚是怎么改的），必要时贴关键代码。

`Summary` 控制在 3 到 5 条以内。每条要具体，不要写空话。

## Changes

说明改动了那些文件，新增了什么函数，这个函数有什么作用，简要说明即可。要有代码树状图（这样比较清晰）。新增的文件要标注新增。

样例：
KnowledgePilot/
├─ apps/
│  ├─ web/
│  │  └─ src/app/
│  │     ├─ page.tsx                       # 开发环境跳过登录，问答请求走 /api/ask
│  │     └─ api/ask/route.ts               # 新增：本地问答演示 API Route
│  └─ api/
│     └─ src/
│        ├─ common/
│        │  ├─ app-env.ts                  # 增加开发开关，修正布尔 env 解析
│        │  ├─ redis.service.ts            # 开发环境 Redis 不可用时降级
│        │  ├─ prisma.service.ts           # 开发环境数据库不可用时降级
│        │  └─ object-storage.service.ts   # 开发环境对象存储不可用时降级
│        └─ modules/
│           ├─ auth/user-access-token.guard.ts # 开发环境可跳过 QA 鉴权
│           ├─ qa/qa.service.ts                # 开发环境直连问答 fallback
│           └─ search/search.service.ts        # 增加开放搜索方法
├─ scripts/dev/start-local-api.cmd          # 新增：本地 API 启动辅助脚本
└─ 启动文件.md                              # 新增：启动排障记录

## 技术内容要求

- 结论要明确，不要模糊说“可能”“似乎”，除非确实不确定。
- 如果有具体文件路径、函数名、变量名，必须写出来。
- 如果有错误代码和期望代码，优先并排说明。
- 如果是转换、解析、布局、路由类问题，要写清楚：
  - 输入是什么
  - 当前输出是什么
  - 期望输出是什么
  - 根因在哪一层
  - 修复应该放在哪个模块


## PR规则补充

如果已经有对应的issue，那么可以在Summary里引用issue，跳过Background，重点在怎么Fix上面

## 代码片段规则

代码片段只放最关键部分。

错误示例：

```python
# 贴一大段无关代码
```

正确示例：

```python
Net__LED1_2_ += LED1['1'], R10['2'], U3['5']
Net__LED1_Pad1_ += LED1['1'], R12['1']
```

然后解释：

```text
`LED1['1']` 同时出现在两条不同 net 中，导致 LED 两端被错误合并。
```

## PR / Issue 输出硬规则

只要用户要求生成 PR、Issue时，最终回复必须是一个完整的 Markdown fenced code block。

硬性要求：

- 回复开头必须直接是四个反引号加 `markdown`
- 回复结尾必须是四个反引号
- 代码框外不能有任何文字
- 不能输出 `Result`
- 不能输出“下面是内容”
- 不能输出解释、总结、补充说明、验证结果
- 不能把内容拆成多个代码框
- 所有内容必须放在同一个代码框里

如果无法满足，必须重新生成，直到最终回复只有一个 Markdown 代码框。
