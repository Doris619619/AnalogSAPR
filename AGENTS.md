# AGENTS.md

## 基本原则

- 优先保证代码正确、清晰、可维护。
- 修改前先阅读相关代码、文档和现有实现，遵循仓库已有的目录结构与编码风格。
- 保持改动范围聚焦，不顺手修改与当前任务无关的内容。
- 不确定需求、边界条件、技术方案或改动影响时，主动向用户确认，不要自行猜测。
- 发现潜在风险、实现冲突或更合适的方案时，及时说明原因并与用户确认。
- 不得删除或覆盖用户已有改动，除非用户明确要求。

## 可维护性要求

- 注释必须是中文为主。
- 每个新增或修改的源代码文件，文件顶部必须添加简短注释，说明文件职责。
- 每个新增或修改的函数、方法、类前必须添加注释，说明用途。
- 注释使用对应语言的规范格式，例如 `#`、`//`、`/** */`。
- 注释应解释职责、关键约束或必要背景，避免逐行复述代码。
- 函数保持职责单一，命名清晰，避免过度复杂的嵌套逻辑。
- 优先复用已有模块和工具函数，避免重复实现。
- 涉及公共接口、核心业务逻辑或行为变更时，应补充或更新测试。
- 不为了抽象而抽象；只有在确实降低复杂度或重复度时才增加封装。
- 不要把过多职责塞进一个文件。页面、组件、服务、DTO、Provider、工具函数和测试应按职责拆分，单个文件只承载一个清晰主题。
- 当文件开始同时处理 UI 展示、状态管理、接口调用、数据转换或业务规则时，应优先拆出局部组件、领域服务或纯函数，而不是继续追加代码。
- API 模块内应按领域组织代码，不要在 `apps/api/src/common`、`shared-config`、泛化 `utils` 中堆放业务逻辑。
- 共享代码只有在至少两个应用稳定复用时才放入 `packages/*`，不要为了“以后可能会用”提前抽共享包。
- 新增依赖、全局状态、通用框架或大型抽象前，应确认确实能降低复杂度；一次性需求优先保持局部实现。


## 提交前检查

- 检查改动是否符合需求。
- 检查是否误改了无关文件。
- 检查新增和修改的源代码文件是否具有文件职责注释。
- 检查新增和修改的函数、方法、类是否具有用途注释。
- 检查是否把多个不相关职责塞进同一个文件；如果文件职责变得模糊，应先拆分再提交。
- 根据改动范围运行对应的 `typecheck`、`lint`、`test` 或 `build`，无法运行时需在最终说明中写明原因。
- 检查是否误提交 `.env`、临时日志、构建产物、缓存目录或本地调试文件。

## Commit 规范

Commit 消息必须使用中文，格式为：

```text
<type>(<scope>): <中文说明>
```

要求：

- `type` 必须使用小写。
- `scope` 必填，使用简短的小写英文。
- 中文说明应简明扼要地说明改了什么以及为什么修改。
- 不添加 `Co-Authored-By` 行。
- 一个 Commit 聚焦一类改动，避免混入无关修改。

允许的 `type`：

```text
feat / fix / chore / refactor / ci / docs / test / perf / style
```

示例：

```text
feat(qa): 增加规则优先匹配逻辑，减少模型回答偏差
fix(auth): 修复验证码过期后仍可登录的问题
docs(repo): 补充仓库协作规范，统一提交和 PR 格式
```

## PR 规范

- 编写 PR 或 Issue 前，必须先阅读并遵循仓库根目录的 `PR_WRITING_RULES.md`。
- PR 标题使用清晰、简短的中文。
- PR 描述必须准确说明改动内容、修改原因、影响范围和验证结果。
- 不夸大改动效果，不隐藏已知风险。
- 如果存在未完成事项、兼容性风险或未运行的检查，必须明确列出。
- PR 中涉及的 Commit 必须符合本文件的 Commit 规范。

## 协作要求

- 需求明确时，直接完成实现、检查和必要验证。
- 操作可能造成数据丢失、覆盖文件、删除内容或较大范围重构时，必须先征得用户同意。
- 完成任务后，简要汇报修改内容、验证结果和仍需关注的事项。
